# Agent Handbook

Shared onboarding layer for AI coding agents in this repository. Keep project facts here. Keep `AGENTS.md` and `CLAUDE.md` thin and agent-specific.

## Project Snapshot

- **Product:** local-first persistent memory system for AI coding agents — captures session history, stores it, and serves reliable semantic recall back to agents. Adds portable export/import and a visual graph UI.
- **Primary users:** solo developers running coding agents (Claude Code, Codex) who want durable cross-session memory; secondarily, the open-source community (public repo).
- **Current maturity:** MVP+ with a live memory layer, shipped on `main`. Foundation (#1–#12, #14): reliable `embed → persist → recall` over MCP, JSONL ingestion, portable export/import, the `abs` CLI, an interactive localhost graph UI, and optional LLM consolidation. Live memory layer (#15–#21): Claude Code hooks for hands-free auto-ingest + context injection (SessionStart baseline/staleness, per-prompt FTS-first), and an optimization loop that turns distilled memory into gated `CLAUDE.md` / auto-memory edits. Plus selective hard-delete across CLI, MCP, and the UI. Public repo.
- **Primary languages and frameworks:** **Node.js ≥22 + TypeScript (ESM)**; SQLite + sqlite-vec + FTS5 storage; transformers.js for local embeddings (pluggable to hosted); the official MCP SDK; esbuild + vanilla TS + three.js (WebGL2 creature renderer) for the UI; **Rust + Tauri 2** for the tray companion (`src-tauri/`). See `docs/adr/`.

## Core Commands

Stack: **Node.js (≥22) + TypeScript (ESM)**, Biome (lint+format), Vitest (tests),
`tsc` (build/typecheck). See `docs/adr/0001-storage-and-embeddings.md` for the
storage/embedding decisions.

- Install: `npm install`
- Test: `npm test` (watch: `npm run test:watch`)
- Lint: `npm run lint` (autofix: `npm run lint:fix`)
- Typecheck: `npm run typecheck` (runs both `tsconfig.json` and the browser-client `tsconfig.ui.json`)
- Build: `npm run build` (`tsc` → `dist/`, then `build:ui` bundles the graph UI → `dist/ui/static/`)
- **Full gate (CI parity):** `npm run check` (lint → typecheck → test). `test` has a `pretest` hook that builds the UI bundle, so the gate is self-contained on a clean checkout. Lint + typecheck also cover `e2e/` (via `tsconfig.e2e.json`), but `check` does **not** run the slow E2E suite.
- **End-to-end system suite (opt-in):** `npm run test:e2e` — drives the built binary, MCP stdio, hooks, optimize/delete, and the UI (headless Playwright) against a throwaway `HOME`/`ABS_HOME`; self-cleaning, offline, real store untouched. One-time: `npx playwright install chromium`. Lives in `e2e/`; see `docs/testing-strategy.md` → "End-to-End System Suite".
- Run a TS entrypoint in dev: `npm run dev`
- Distill a session into lessons (opt-in, needs `ABS_LLM_BASE_URL`+`ABS_LLM_MODEL`): `abs consolidate [--session N] [--dry-run] [--force]`
- Register the Claude Code memory hooks (auto-ingest + context injection; opt-in, idempotent, backup-first): `abs install-hooks`. The registered hooks call `abs hook <session-end|session-start|user-prompt-submit>` internally — non-fatal/timeout-bounded, never invoked by hand. See ADR 0004/0005. **SessionEnd auto-ingest is scoped to the CURRENT session only** (`ingestSingleSession` on the payload's `transcript_path`) — it never walks the whole `~/.claude/projects` tree, so a fresh/reset store is never silently back-filled with unrelated projects' history (#62).
- **Multi-harness support (#65–#72) — five qualifying harnesses.** Every harness is wired the same two ways: `abs install-hooks --harness <id>` (lifecycle wiring) + `abs setup --harness <id>` (hooks + MCP), reversed by `abs uninstall --harness <id>`. With **no flag**, all three target the Claude Code reference adapter. `abs status` lists each harness's `installed` (`adapter.detect()`) + `parity` (`adapter.qualifies().ok`) so you know which `--harness <id>` to run. Supported ids: `claude-code`, `codex`, `gemini`, `copilot`, `opencode`.
  - **Architecture:** all harness-specific concern lives behind the `HarnessAdapter` contract (`src/harness/types.ts`); the core (store/recall/embed/optimize) never names a harness. `defaultRegistry()` (`src/harness/index.ts`) holds the five adapters; each is assembled from reusable `src/harness/capabilities/` (lifecycle installers, MCP registrars, session resolvers, transcript sources). The harness namespace (`codex:`/`gemini:`/`copilot:`/`opencode:`; Claude stays bare) is applied ONCE at the `dispatch.ts` chokepoint. Per-harness mechanics + the parity policy are in **ADR 0012**.
  - **How to add a harness:** implement `HarnessAdapter` in `src/harness/adapters/<id>.ts` (reuse a `capabilities/` installer/registrar/resolver, or add a new one), map the harness's native events onto the `capture`/`recall`/`guard` moments in `eventMap`, return `{ ok: true }` from `qualifies()` only when all four parity pillars hold (capture, recall, MCP, stable session id + readable transcript), then register it in `defaultRegistry()`. Add the namespace at the `dispatch.ts` chokepoint, never in the core. Mirror an existing adapter's test file.
  - **Codex specifics (#67):** hooks live in `~/.codex/config.toml` under a managed `[hooks]` block (TOML, sentinel-delimited); capture is the `Stop` event (no `SessionEnd`). **Trust gate:** Codex SILENTLY skips `config.toml`-managed hooks in untrusted projects, so `abs install-hooks --harness codex` warns when the current project's `[projects."<cwd>"] trust_level` is not `"trusted"`. Accepted degradations: MCP `recall` is cwd-scoped (no hook payload → no harness); `abs project` needs explicit `--session codex:<uuid>` (no session-id env var). Capture is prose-only for now (tool-anchor extraction is a tracked follow-on).
- Reverse of `abs setup` (remove hooks + unregister the MCP server; idempotent, backup-first): `abs uninstall`. Preserves the memory store by default; `--purge` hard-deletes it (confirmed unless `--yes`). Prints the final `npm uninstall -g` to run yourself.
- **Explicit, opt-in historical ingest** of past transcripts (the deliberate counterpart to the per-session SessionEnd auto-ingest): `abs ingest [--apply] [--all | --project <slug> ...] [--dir PATH]`. Preview-only by default (lists projects + how much is new, writes nothing); `--apply` ingests but **requires a selector** (`--all` or `--project`), so a full back-fill is never accidental (#62). The per-file byte cursors live in the store (`kv_meta` `ingest:cursor:<absPath>`), so resetting the store also clears them — a subsequent ingest re-reads from offset 0.
- Turn distilled memory into gated `CLAUDE.md` / auto-memory edits (preview → approve → apply; never automatic): `abs optimize [--project PATH] [--limit N] [--apply] [--candidate ID] [--yes]`. See ADR 0006. **Project-scoped** (#135): candidates draw only on the consolidated memory of the optimized project (`projectSlug(--project|cwd)`, the same label recall/ingest use) and write only into that project's targets — another project's decisions never leak into this repo's `CLAUDE.md`. Candidate ids are content-addressed (stable per evidence), so `apply <id>` can never bind a different candidate than the one previewed. **Curated (#146, ADR 0016):** a $0 heuristic floor and an opt-in LLM judge together drop operational trivia (install one-offs, action/event logs, external-tool config) before promotion; the summary reports how many items were "held back as low-durability". Dropped items stay in the store, still recallable. After the apply loop it advances the two kind/project optimize cursors per the keep-set partition (ADR 0017) — the SAME authority the cadence uses, so manual and automatic runs advance identically (a bare preview / all-declined run advances nothing).
- **Auto-distill cadence (#138/#148, ADR 0017) — internal, on by default when an LLM is configured:** `abs maintain --auto` drives `consolidate → generateOptimizations → auto-apply auto-memory only` behind a dedicated `.cadence.lock`, advances the kind/project optimize cursors, and records token observability. It is **spawned detached** by SessionEnd when a session is cadence-due (LLM configured **and** `ABS_AUTO_DISTILL` not opted out **and** the just-ended session ≥ `DISTILL_MIN_OBS`, default 25) — the hook makes no LLM/network call. **Scope (deliberate):** it promotes lessons to the project's auto-memory entry (`consolidated-lessons.md` + an additive `MEMORY.md` pointer, #140) and **never** writes the git-tracked `CLAUDE.md`; skipped decisions keep nagging via their own un-advanced cursor until a manual `abs optimize`. Opt out with `ABS_AUTO_DISTILL=0`; the first auto-run announces itself once in the SessionStart injection (per-session cost + the opt-out var). Benign no-op with no LLM or a held lock. `maintain` without `--auto` prints usage and exits 0.
- **Staleness is two independent signals** (ADR 0017, supersedes the single optimize cursor of #16/#21), both project-scoped: (1) **raw-pending** = a session-level anti-join (raw turns whose session has NO `source='consolidate'` row — `countUnconsolidatedRawTurns`/`countUnconsolidatedSessions`; an `EXISTS`, never an id cursor, so it can't strand a session); (2) **consolidated-pending** = lessons-pending + decisions-pending counted above the two kind/project cursors `optimize:lesson:<slug>` / `optimize:decision:<slug>`. The cadence auto-clears lessons; decisions persist until a manual promote. No consolidate cursor key exists.
- Selectively hard-delete memories (IRREVERSIBLE, preview-only without `--apply`): `abs forget [--ids a,b,c | --session N | --project NAME | --null-project | --search "q" [--limit N]] [--apply] [--yes]`. Export first — no backup. See ADR 0008.
- **Portable export/import is crash-atomic** (`abs export <path>` / `abs import <path> [--mode merge|replace]`, #157): export streams to a temp file and renames only after the full write succeeds, so a failed export never truncates an existing artifact; `import --mode replace` wipes-and-loads inside one `store.transaction(...)` (same destructive-op philosophy as ADR 0008), so a mid-import failure rolls back to the pre-import state instead of leaving an emptied store. Import reconciles the artifact header counts against the rows actually parsed and rejects a short/truncated artifact (`export artifact truncated: ...`) instead of reporting false success. `replace` is a **true reset**: beyond sessions/observations it also clears the ingest-state `kv_meta` (cursors, harness cwd/id caches, degraded/rebuild flags) so the next ingest re-reads from offset 0 (the reset contract promised under `abs ingest` above), while preserving `session-project:*` bindings (explicit user decisions, not tied to the deleted data).
- Confirm/skip the CURRENT session's project (writes the decision binding of ADR 0010, applied at next ingest). The project is ALWAYS the session's folder (cwd) — there is no custom name: `abs project [--cwd | --skip] [--session ID] [--yes] [--json]`. Session id resolves from `CLAUDE_CODE_SESSION_ID` (NOT latest-transcript-by-mtime — subagents pollute the dir), `--session` overrides. `--cwd` files it under the folder (and reverts a prior skip); `--skip` hard-deletes any already-stored observations for the session → requires `--yes` (ADR 0008). No-arg prints status.
- MCP tools exposed to agents: `recall`, `remember`, `memory_status`, `optimize`/`apply` (gated edits), `forget_preview`/`forget` (two-phase selective delete: preview mints a single-use handle, `forget` consumes it — IRREVERSIBLE; ADR 0008), and `set_session_project` (#52: decide whether this session is saved — `action=skip|include`; the project is always the folder, so there is no name to choose; pass `session` from the SessionStart notice, `CLAUDE_CODE_SESSION_ID` fallback; `skip`'s hard-delete of stored observations needs `confirmDelete=true`, the MCP analog of `abs project --skip --yes`; ADR 0008/0010). The UI write path (delete) is gated by the localhost CSRF/Origin controls in ADR 0007.
- SessionStart injects, besides the baseline, a **memory notice** (#52, ADR 0010): the project is always the session's folder, so instead of asking which project, it tells Claude to let the user know — once — that this session is being saved under that folder, and that they can ask to leave it out (recorded via `set_session_project` action `skip`). $0/offline (`ensure:false`), non-fatal, suppressed once a binding exists or when the payload has no session id. Soft by nature; UserPromptSubmit reinforces it on the session's first prompt (a one-time `notice-shown:<sid>` flag, suppressed when the session is already skipped).
- **Recall is project-scoped by default** (#47): per-prompt injection, the MCP `recall` tool, and `recall`/`recallFts` filter to the current session's project so memory from other projects never leaks in. `ABS_RECALL_SCOPE=global` opts back into store-wide recall. The scope label is resolved (`src/recall/scope.ts` `resolveRecallProject`) from the **stored** label — `set` binding → stored `sessions.project` row → `projectSlug(cwd)` (canonical, byte-identical to what ingest now derives from each line's `cwd`) → `basename(dirname(transcript_path))` fallback when no cwd — never a freshly recomputed slug that diverges, so worktree/dual-encoded-dir sessions don't silently starve. The injected fence header names the active project. Store legs (`searchFts`/`knn`) apply the filter in SQL (JOIN `sessions` / `rowid IN`), never a TS post-filter.

## Architecture Notes

- **Runtime shape:** local daemon/CLI + MCP server + localhost web UI + Claude Code lifecycle hooks. Single-user, local-first.
- **Key components:** session ingestion (JSONL) → embedding layer (local-default, pluggable, with hosted retry/backoff) → embedded vector + keyword store → recall API over MCP → optional LLM consolidation (session → lessons) → export/import → localhost graph UI. **Live memory layer on top:** hooks auto-ingest each session and inject relevant memory back (SessionStart + per-prompt FTS-first, no cold-load); the optimization loop turns distilled memory into gated `CLAUDE.md`/auto-memory edits; selective hard-delete prunes by id/session/project/search across CLI/MCP/UI.
- **State/data boundaries that are easy to break:** the index lifecycle (embedding → persistence → rebuild → recall). The reference tool we are replacing failed precisely here — indexing not invoked, indexes never persisted/rebuilt. Treat the **embed→persist→recall** path as the critical, test-first surface.

## Change Boundaries

- The memory store schema and the index lifecycle are the highest-risk areas — change with tests. **The Tray companion reads the schema directly** (`src-tauri/src/lib.rs` queries `observations`/`sessions` and the columns `id`/`created_at`, read-only via `rusqlite`); a migration that renames those identifiers must update the Rust queries too (no compiler links them). Its dbPath precedence mirrors `config.ts` `loadConfig` — keep both in step.
- The MCP tool contract is a public interface — version it deliberately.
- **LLM consolidation** (`src/consolidate/`) writes lessons ONLY through the indexer (recallability invariant), is opt-in/idempotent, and treats ingested transcripts as untrusted (output constrained to the lesson/decision schema). See ADR 0003.
- **Hooks** (`src/hooks/`) must stay non-fatal + timeout-bounded — any failure exits 0 and never blocks a Claude Code session (ADR 0004). The per-prompt injection path must stay FTS-first (no embedding cold-load) within the ADR 0005 latency baseline.
- **Write-to-real-files paths are guarded and gated.** `src/optimize/` edits only `CLAUDE.md` + auto-memory with per-candidate approval + backup/atomic/rollback + a fail-closed `user|feedback` guard (ADR 0006). `src/delete/` is irreversible hard-delete behind preview→pinned-handle→execute (ADR 0008). The UI write-path adds CSRF token + Host/Origin allowlist + handle confirmation (ADR 0007). Change these with their dedicated tests; never make them automatic.
- **The auto-distill cadence + optimize cursors are sensitive — change with tests** (ADR 0017). The cadence (`src/maintain/`) is the ONE automatic writer, and even it only writes the project's **auto-memory** (`consolidated-lessons.md` + `MEMORY.md` pointer) — never the git-tracked `CLAUDE.md`. The kind/project optimize-cursor advance (`advanceOptimizeCursorsAfterApply` in `src/optimize/run.ts`, wired into both the cadence and manual `cmdOptimize`) is keep-set driven: advance a kind cursor IFF its pending-valid set is empty, so a wrong rule strands or double-swallows observations. `applyApprovedCandidate` advances **no** cursor on its own. The two-signal staleness (`src/hooks/staleness.ts`) uses a session-level anti-join for raw-pending — never reintroduce a consolidate cursor. The dedicated `.cadence.lock` (`src/store/cadence-lock.ts`) is NOT the rebuild lock; its `acquire` must return ownership atomically from the `wx` create (no pre-acquire `isLocked` read — that is the double-billable-LLM TOCTOU). Migration v6's `idx_observations_session_source` keeps the anti-join + filtered counts $0/offline.
- **Ingest is decision-aware** (ADR 0010). A session's project is its folder — derived from each transcript line's `cwd` (`projectSlug(entry.cwd)`, falling back to the storage dir name only for lines with no cwd), so subagent transcripts (`.../subagents/`) and dual-encoded dirs resolve to the right project instead of accidental buckets. An intentional binding in `kv_meta` (`session-project:<sessionId>`) can override: `skip` excludes the session entirely (no row, cursor still advances; reconciles an already-ingested session); `--cwd`/`include` files it under its folder (reverting a skip). No binding → the cwd-derived project. The binding is at-least-once (applied whenever ingest next runs — `SessionEnd` is not guaranteed on abrupt kill, validated by spike #49). Writers: CLI `abs project` (#51) / MCP tool (#52); consumer: project-scoped recall (#47).
- **Global brain** (curated cross-project memory, recalled in *every* project alongside the project brain). A reserved session (`external_id = project = '__global__'`, `src/global.ts`; collision-free — ingest never produces that label). Authoring is **user-initiated only**: write to it (MCP `remember` `scope:"global"` / `rememberAction`; `abs remember --global "..." [--kind K]`) or promote an existing memory into it (MCP `promote` / `abs promote <id>`) ONLY when the user explicitly asks — never on your own initiative. Promote defaults to *moving* the observation into the global session (`moveObservationToSession`); with `as`/`--as "<text>"` it instead files a curated COPY of exactly that text via `indexer.write` (`promoteAction` in `src/global.ts`), keeping the original in its project — so you promote only the reusable wording and leave project-specific or sensitive detail behind. The curated copy inherits no metadata (only `{promotedFrom}`) and is a 2nd recallable item by design. Recall ORs the global session into the project filter: `searchFts`/`knn`/`recallFts` take `includeGlobal` (the hooks pass `true`), and global hits are tagged `🌐global` in the injected block. Relevance-based like the project brain — no always-inject tier. `abs forget --global` prunes it (gated, ADR 0008); `abs status` reports `globalObservations`; `listProjects` excludes the sentinel.
- **The verifiable-memory layer is fail-open and read-only against ground truth** (ADR 0009). The PreToolUse guard (`src/hooks/pre-tool-use.ts`) only reads `tool_input` (never executes it), point-queries the graph read-only, warns by default (`ABS_GUARD_MODE=block` opts into deny), and degrades to silent-allow with no graph. Self-healing marks anchors `stale`, never deletes. `fact_anchors` lives in the index lifecycle (`embed → persist → recall`) — change with migration tests. Freshness seals propagate **turn-scoped**: a Claude code turn emits prose and its Edit/Write as separate JSONL lines (separate observations), so ingest copies the edit's **file-level** anchors onto the sibling prose observation — keyed on the transcript's per-turn `message.id`/`requestId` — so the narrative that recall actually surfaces carries the `✓verified`/`⚠stale` seal, not only the edit-summary obs (`propagateFileAnchors`, `src/ingest/ingest.ts`; deduped, never session-wide).
- **The PreToolUse guard has two lenses** (#48 Phase A). (1) **Code duplication** (graph, ground-truth) — BLOCK-eligible under `ABS_GUARD_MODE=block`. (2) **Decision surfacing** (memory) — on an Edit/Write it recalls `decision`/`lesson` observations related to the touched file/symbols (project-scoped #47 FTS, $0/offline) and shows them as a **warn-only** note ("keep your change consistent"); it surfaces, never judges contradiction, and never blocks (even in `block` mode). Duplication has priority — a decision note never rides inside a deny payload. Only the block-eligible duplication lens is gated by the O3 eval (`guard-eval.ts`, TP ≥ 30% / FP < 1 per 10 actions); decision surfacing is warn-only so it ships ungated. **Structured decisions + deterministic violation matching (#48 Phase B) and an opt-in LLM-judge (#48 Phase C) are follow-ups** — Phase A only surfaces.
